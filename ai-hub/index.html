Everything here is oriented around practical, local-first tooling. The knowledge graphs are real open-source benchmarks. The skills run in Claude Code today. Wyltek Studio runs on your own GPU.

graphify — Knowledge Graph Builder

pip install graphifyy  ·  CLI: graphify

Build queryable knowledge graphs from codebases, documentation, or any text corpus. Produces an interactive HTML viewer, GraphRAG-ready JSON, and a GRAPH_REPORT.md — in a single command.

pip install graphifyy    # two y's — PyPI namespace
graphify .               # build from current directory
graphify . --mode deep   # thorough extraction
graphify --mcp           # start MCP server mode

What graphify produces

graph.json — GraphRAG-ready: nodes, edges, communities, metadata. Query it directly or via MCP.

graph.html — Self-contained interactive viewer. Force-directed layout, search, filter by community or type, click to inspect.

GRAPH_REPORT.md — Top communities by size, key entities, notable relationships. Use as a CLAUDE.md appendix.

Knowledge Graphs — Build Once, Query Forever

Naive RAG retrieves text chunks by semantic similarity. Knowledge graphs model the actual relationships between entities — calls, imports, dependencies, implementations — and return the minimal relevant subgraph for any question. The result is dramatically fewer tokens with better cross-file accuracy.

How It Works

Four-stage pipeline

1. Entity Extraction — Language-aware parsers (Rust, TypeScript, C, Python, Go, Markdown) extract functions, classes, modules, types, CLI commands, and API endpoints.

2. Relationship Mapping — Entities are connected: calls, imports, implements, extends, depends_on. Cross-file and cross-repo edges are resolved.

3. Community Detection — The Louvain algorithm groups tightly related entities into communities — these map naturally to architectural subsystems.

4. Output Generation — Interactive HTML viewer, GraphRAG JSON, GRAPH_REPORT.md. Optional Neo4j export.

graph-routing skill

A Claude Code skill that sits in front of every cross-repo and architecture question. When triggered, it:

  1. Detects the question shape (cross-repo, dependency trace, architecture, debug-with-symptoms)
  2. Routes to the appropriate graph in ~/.claude/graphs.json
  3. Runs a targeted BFS query against that graph
  4. Returns the relevant subgraph as context
  5. Proceeds to answer — no grep/glob needed

Separate graphs for separate domains keeps queries fast and prevents cross-contamination. The registry lets the routing skill pick the right graph automatically.

Quick Start

# Install (note: two y's in the package name)
pip install graphifyy

# Build a graph from your project
graphify ~/projects/my-project --mode deep

# Outputs:
#   graph.json      — GraphRAG-ready, for AI agents
#   graph.html      — Interactive viewer, open in any browser
#   GRAPH_REPORT.md — Community summary, key entities

# Start the MCP server (Claude Code native integration)
graphify --mcp --graph ./graph.json --port 3100

# Watch mode — auto-updates as files change
graphify . --watch

Wyltek GPT — Local Chatbot for Your Phone

Mobile-first browser terminal for local Ollama models, with a 25-converter file pipeline, PWA install via Tailscale, Android share-target, and skinnable themes. Built to be a daily-driver local inference UI you can install on your phone and use to convert files between formats without ever leaving the browser.

GitHub — toastmanAu/wyltek-gpt

What It Does

Chat with Any Local Ollama Model
Model dropdown, streaming output, per-request system prompt assembled with live host facts (current date, OS, hostname) so models don't hallucinate about your environment.
Convert Between 25 Formats
A unified file tray that routes each conversion to the right tool: markitdown for "anything → md", libreoffice for office → pdf, ffmpeg per-target for video (with faststart for browser playback), whisper for audio → text, pandoc / weasyprint / calibre for documents and ebooks, vips / cwebp / rsvg-convert for images.
Mobile Share-Target
On Android, long-press any file in any app, hit Share → wyltek-gpt, and the conversion tray pre-loads with target formats filtered to what's actually possible from the file you sent.
Skinnable + Indicatif Spinners
Five built-in themes (tokyo-night, catppuccin-mocha, nord, classic-green, custom). Settings drawer lets you swap themes and pick from nine Unicode spinners (braille, dots, line, bar, arrow, bounce, pulse, moon, clock).

The Converter Pipeline

Documents & Ebooks
markitdown, pandoc, weasyprint, calibre, libreoffice. Anything → markdown via markitdown; office → pdf via libreoffice; html → pdf via weasyprint; epub ↔ mobi/azw3 via calibre.
Images
vips for resize and format-conversion, cwebp for high-quality WebP, rsvg-convert for SVG → raster. Per-converter quality params (quality, lossless, dimensions).
Audio & Transcription
ffmpeg for audio re-encoding with bitrate control. whisper for audio/video → text with selectable model size per request (tiny / base / small / medium / large).
Video
ffmpeg with target-aware presets. MP4 outputs use +faststart for browser-streamable playback. CRF, bitrate, fps, and resolution params per target.
Missing binaries are soft-skipped At startup, each converter is checked for its binary. Missing tools are logged as warnings and dropped from the enabled list shown to the model and UI — so the menu always reflects what's actually installed.

Install on Your Phone

Service workers — which power PWA install and Android Web Share Target — only register over HTTPS or localhost. Plain LAN IPs like 192.168.x.x:8000 silently fail. Tailscale Serve fixes this in one command, with a *.ts.net cert visible only inside your tailnet.

# On the host running uvicorn
uvicorn backend.app:app --host 127.0.0.1 --port 8000

# Expose with HTTPS via Tailscale
tailscale serve --bg --https=443 http://127.0.0.1:8000
tailscale serve status   # prints the URL to open on your phone

Per-platform notes

Android — Chrome surfaces "Install app" automatically once the service worker registers. After install, wyltek-gpt appears in every app's share sheet.

iOS — Safari → Share → "Add to Home Screen". iOS does not support Web Share Target; the workaround is an iOS Shortcut that POSTs files to /api/upload.

Security Model

Fixed argv, never shell-assembled
The model never builds shell strings. Each converter has a fixed argv template in config.yaml. User-supplied params (quality, bitrate, model size) are validated against a per-converter whitelist before the subprocess runs.
No shell=True, ever
All subprocess calls are subprocess.run([...], shell=False). There is no code path that constructs a shell command from any input — model output, user input, or filename.
Sandboxed workspace
All file paths are resolved with Path.resolve() and checked to be inside the session workspace root before any read, write, or subprocess. Filenames at the upload boundary are normalised with Path(name).name to block path traversal.
Local-first by default
Default bind is 127.0.0.1 — nothing on the LAN. Going phone-accessible is an explicit opt-in via Tailscale, where the URL is only reachable inside your tailnet.

Research Pipeline & Findings API

An automated research pipeline where AI agents submit topics, a processor fetches source material and generates structured findings, and results are immediately available via REST API — and injected into Claude Code sessions via the RAG hook.

Browse Research Findings API Reference

How the Pipeline Works

1. Submit a Research Task
Any agent (or a shell script) writes a markdown task file to the research inbox. The task specifies a goal, seed URLs, priority, and requester. Claude Code can submit tasks mid-session without interrupting work.
2. Autonomous Processing
The research processor fetches seed URLs, extracts content, and analyses it with an LLM (cloud or local Ollama fallback). Findings are written as structured markdown within minutes.
3. RAG Ingest
Findings are automatically ingested into the RAG system via the REST ingest endpoint. They become immediately queryable by Shannon and Kernel — and auto-injected into Claude Code sessions via the UserPromptSubmit hook.
4. Website Sync
Findings sync to this site every 6 hours. Also pushed to an Obsidian vault for human-readable offline access. Git-backed, versioned, linkable.

REST API

# List recent findings
GET /api/findings?limit=20&topic=blockchain

# Get a specific finding
GET /api/findings/:id

# Ingest a new finding (from agents or scripts)
POST /api/ingest
Content-Type: application/json
{
  "source": "research-task-id",
  "text": "Finding content...",
  "title": "Title of the finding"
}

# Query the knowledge base
POST /api/query
Content-Type: application/json
{
  "query": "CKB transaction fee estimation",
  "agent": "kernel",
  "k": 3
}
No auth required for read endpoints GET endpoints are open — any agent, browser, or script can browse and query findings. Ingest requires the API to be running on your local network (it is not a public write endpoint).
