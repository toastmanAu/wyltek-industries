Technical Resolution Mechanisms\n\nA browser needs to translate a blockchain domain into content. Here are all viable approaches, ranked by decentralization:\n\n### A. Browser Extension (Intercepting URL Requests)\n**How it works**: Extension uses `chrome.declarativeNetRequest` or proxy settings to intercept navigation to blockchain TLDs. Queries blockchain (directly or via RPC) for domain records. Redirects browser to resolved IP/IPFS gateway.\n\n**Pros**: No system-level changes, easy to install, can verify proofs locally.\n**Cons**: Extension-only (not native), Chrome Manifest V3 restrictions limit background processing, each browser needs its own extension.\n\n**Examples**: Unstoppable Domains extension, B-DNS extension, PeerName extension.\n\n### B. Local DNS Proxy\n**How it works**: A daemon runs locally, acts as a recursive DNS resolver. For blockchain TLDs, it queries the chain (via full node, SPV, or indexer). For everything else, forwards to upstream DNS.\n\n**Pros**: Works for all applications (not just browsers), system-wide resolution, can integrate DNSSEC.\n**Cons**: Requires local software installation, potentially runs a light client.\n\n**Examples**: hnsd (Handshake), ncdns (Namecoin). Both use libunbound as the recursive resolver with a custom authoritative backend.\n\n### C. HTTP/HTTPS Gateway\n**How it works**: A web server resolves blockchain domains and serves content. User accesses `domain.eth.limo` or `domain.crypto.unstoppable.io`. Gateway reads chain, fetches IPFS/content, returns HTML.\n\n**Pros**: Zero-install, works in any browser.\n**Cons**: Centralized single point of failure/censorship, gateway operator sees all traffic, no trustless verification.\n\n**Examples**: eth.limo (ENS), resolver.unstoppable.io, HNSDNS.com.\n\n### D. DNS-over-HTTPS (DoH) with Blockchain Backend\n**How it works**: A DoH server (RFC 8484 compatible) resolves blockchain names. Browser or OS configured to use the DoH endpoint. Server runs a light client or indexer internally.\n\n**Pros**: Uses standard DNS infrastructure, browsers already support DoH, encrypted transport.\n**Cons**: Still relies on a remote server (though could self-host), DoH spec doesn't change the authoritative/recursive split.\n\n**Examples**: HNSDoH (`hnsdoh.com`), Unstoppable's `resolver.unstoppable.io/dns-query`, Brave's integration.\n\n### E. Service Worker In-Browser\n**How it works**: A service worker intercepts fetch events, resolves blockchain domains, and proxies content. Could use an in-browser light client (WASM).\n\n**Pros**: No extension needed (once registered), runs in any modern browser.\n**Cons**: Service workers can only intercept requests within their scope (same origin), cannot intercept top-level navigation to new domains. **This makes service workers unsuitable for DNS-like resolution of arbitrary domains.**\n\n### F. Native Browser Integration\n**How it works**: Browser natively recognizes blockchain TLDs and resolves them. May use built-in light client or configured RPC endpoint.\n\n**Pros**: Seamless UX, no extensions needed.\n**Cons**: Requires browser vendor adoption.\n\n**Examples**: Brave (ENS + Unstoppable via Infura), Beacon (Handshake via built-in hnsd/hnsquery).\n\n---\n\n## 4. Key considerations:\n\n- **Bundle size**: Not publicly documented, but Rust WASM light clients (cf. For DNS lookups, you'd register a type script hash as a filter pattern and receive matching cells.\n- **Verification**: The light client can verify Merkle proofs for cells without trusting a gateway. This is the key advantage -- truly trustless resolution.\n- **IndexedDB quotas**: Browsers limit IndexedDB storage (varies by browser, typically 50MB-few GB). Header chain storage could be significant.\n\n**Practical architecture for a CKB DNS extension:**\n1. Extension loads WASM light client on install\n2. Syncs headers in background (progressive, could checkpoint)\n3. On domain lookup: compute type script hash from domain name, query light client for matching live cell\n4. Light client requests compact block filter matches from full nodes, verifies Merkle proofs\n5.
