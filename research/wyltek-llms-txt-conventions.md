### Summary

The `llms.txt` convention, proposed by Jeremy Howard in 2024, aims to standardize how websites provide concise, LLM-friendly information to AI agents at inference time. Unlike `robots.txt` (for access control) or `sitemap.xml` (comprehensive page listing), `llms.txt` offers a curated, structured Markdown overview of a site's capabilities and key documentation, specifically designed for LLM consumption. It emphasizes linking to clean Markdown versions of relevant web pages (`.md` appended to the URL) to overcome the challenges of parsing complex HTML.

A high-quality `llms.txt` for a multi-page agentic web app like Wyltek Studio's `open-palette` should follow a specific Markdown structure: an H1 title, a blockquote summary, optional introductory text, and then one or more H2-delimited sections containing lists of relevant hyperlinks. The "Optional" H2 section is a notable feature, allowing agents to skip less critical information when context windows are limited. Examples from PixiJS and Anthropic demonstrate how this structure can be adapted for varying levels of complexity, from focused documentation to extensive API references across multiple languages.

### Key Findings

1.  **Canonical Structure of `llms.txt`:**
    *   **Location:** The file should be located at the root path `/llms.txt` of a website, or optionally in a subpath.
    *   **Format:** Markdown, designed to be both human-readable and programmatically parsable.
    *   **Required Sections (in order):**
        *   `# H1 Title`: The name of the project or site (e.g., `# Wyltek Studio: Open-Palette`). This is the only strictly required section.
        *   `> Blockquote Summary`: A short summary of the project, containing key information necessary for understanding the rest of the file.
    *   **Optional Sections (in order):**
        *   `Zero or more markdown sections`: These can be paragraphs, lists, etc. (but *not* headings), providing more detailed information about the project or how to interpret the linked files.
        *   `Zero or more H2-delimited "file lists"`: Each H2 header introduces a list of relevant URLs.
            *   Each item in these lists must be a Markdown hyperlink `[name](url)`.
            *   Optionally, a colon `:` followed by notes about the file can be included after the hyperlink.
    *   **"Optional" H2 Section:** An H2 header specifically titled `## Optional` (or similar, though `Optional` is the convention) signifies that the URLs listed under it can be skipped by an LLM agent if a shorter context is needed. This is crucial for managing context window limitations.

2.  **Differences between Minimal `llms.txt` and `llms-full.txt` (Context Files):**
    *   The `llms.txt` specification itself defines a single file.
    *   The distinction between "minimal" and "full" refers to derived context files generated by tools like `llms_txt2ctx`.
    *   `llms-ctx.txt` (minimal): Generated by excluding the content linked under the `## Optional` section.
    *   `llms-ctx-full.txt` (full): Generated by including all content, including that linked under the `## Optional` section.
    *   This mechanism allows LLM agents to dynamically adjust the amount of information ingested based on available context window size or user intent.

3.  **Notable Examples from Multi-Route Web Apps:**
    *   **PixiJS (`https://pixijs.com/llms.txt`):**
        *   Uses a single `## Table of Contents` H2 section to list all relevant documentation links.
        *   All linked URLs point to `.md` versions of their respective documentation pages (e.g., `https://pixijs.com/8.x/guides/getting-started/ecosystem.md`).
        *   Demonstrates a clean, straightforward approach suitable for a well-structured technical documentation site. It does not utilize the `## Optional` section.
    *   **Anthropic (`https://docs.anthropic.com/llms.txt`):**
        *   Highly comprehensive, showcasing how to manage extensive and multi-language documentation.
        *   Uses H2 headers for major categories like `## English`, `## Messages`, `## Managed Agents`, `## Admin`, `## API Reference`.
        *   Within these H2 sections, it lists numerous `[Title](URL.md)` links, often with brief descriptions.
        *   Notably, the `## English` section acts as a parent, with further H2s (`### Messages`, `### Managed Agents`, etc.) nested conceptually, though Markdown only supports H1-H6. Anthropic uses H2 for top-level categories and then lists of links under them, effectively using H2s as general section headers for different functional areas.
        *   The `API Reference` section is particularly detailed, providing links to API documentation for various programming languages (e.g., `(cli)`, `(csharp)`, `(Python)`).
        *   This example demonstrates how to structure `llms.txt` for a complex platform with diverse features and API surfaces.
        *   *(Note: The provided Anthropic source content was truncated due to a fetch error, but enough was available to discern its structure and patterns.)*

4.  **Conventions around Versioning, Regeneration Cadence, or `robots.txt` Interaction:**
    *   **Versioning/Cadence:** The `llmstxt.org` specification does not explicitly define conventions for versioning the `llms.txt` file itself or its regeneration cadence. The "Published" date on the spec page (September 3, 2024) suggests updates are tied to the content. Best practice would likely align with major documentation updates or feature releases.
    *   **`robots.txt` Interaction:** `llms.txt` is designed to *complement* `robots.txt`.
        *   `robots.txt` dictates what automated tools *can* or *cannot* access on a site.
        *   `llms.txt` provides *context and curated information* for content that LLMs *are allowed* to access. It helps LLMs understand and utilize the allowed content more effectively, rather than restricting access.
    *   **`sitemap.xml` Comparison:** `llms.txt` is not a substitute for `sitemap.xml`. `sitemap.xml` lists all indexable human-readable pages, often too numerous and not in LLM-friendly formats. `llms.txt` is a curated, LLM-optimized subset, potentially including external links and `.md` versions of pages.

### Questions Answered

*   **Current canonical structure of `llms.txt`:** Detailed in Key Finding #1. It's a Markdown file with a specific H1, blockquote, optional intro, and H2-delimited file lists, with a special `## Optional` section.
*   **Differences between minimal `llms.txt` and `llms-full.txt`:** Detailed in Key Finding #2. These refer to derived context files, with the `## Optional` section being the differentiator.
*   **Notable examples from products with multi-route web apps:** PixiJS and Anthropic provide excellent patterns for different scales of multi-route applications, as detailed in Key Finding #3. Anthropic's approach is particularly relevant for complex platforms.
*   **Any conventions around versioning, regeneration cadence, or `robots.txt` interaction:** Detailed in Key Finding #4. No explicit versioning/cadence, but it complements `robots.txt` for LLM inference.

### Gaps / Follow-up

1.  **Explicit Versioning:** While the spec doesn't mandate it, consider adding a `Version:` or `Last-Updated:` field within the `llms.txt` file (perhaps in the initial descriptive text) to help agents understand content freshness.
2.  **Regeneration Cadence:** Establish an internal policy for `open-palette` on when to regenerate/update `llms.txt` (e.g., with every major release, weekly, on-demand).
3.  **Tooling for `.md` Generation:** Investigate or develop internal tooling to automatically generate and maintain `.md` versions of `open-palette`'s web pages, especially for dynamic content or API documentation. This is a critical component of the `llms.txt` ecosystem.
4.  **Handling Dynamic Content/APIs:** How should `llms.txt` best represent highly dynamic API endpoints or user-specific content? The Anthropic example shows extensive API links, but `open-palette` might have more interactive or personalized elements. Consider linking to API specifications (e.g., OpenAPI/Swagger in Markdown format) or general API usage guides rather than every single endpoint.
5.  **Multi-language Support:** If `open-palette` is to be localized, how will the `llms.txt` handle multiple languages? Anthropic provides separate H2 sections for each language, linking to language-specific documentation. This pattern should be considered.
6.  **Testing with LLM Agents:** Once drafted, `open-palette`'s `llms.txt` should be thoroughly tested with various LLM agents to ensure it effectively answers questions and guides interactions.

### Relevant Code/API Snippets

#### Canonical `llms.txt` Format Example (from `llmstxt.org`)

```markdown
# Title

> Optional description goes here

Optional details go here

## Section name

- [Link title](https://link_url): Optional link details

## Optional

- [Link title](https://link_url)
```
