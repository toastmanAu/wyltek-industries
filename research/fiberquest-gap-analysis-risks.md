# Research: fiberquest-gap-analysis-risks

**Date:** 2026-03-05  
**Status:** SYNTHESIS  
**Seeds:** local findings + MEMORY.md

---

As a technical architect and project advisor, I've synthesized the provided research findings and project ground truth for FiberQuest. The project leverages a robust CKB foundation, but critical integration and verification steps remain for the hackathon MVP.

### Executive Summary

FiberQuest is poised to deliver an innovative hackathon experience by integrating retro gaming with Nervos Fiber micropayments. The core infrastructure, including operational Fiber nodes on `ckbnode` and `N100`, CKBFS for content, and the DOB Minter, is largely functional. Crucially, specific game RAM addresses have been identified for key titles, enabling direct game state interaction. However, the exact JSON-RPC formats for Fiber, the absence of a Node.js Fiber client library, and the secure management of private keys for the sidecar are significant unverified technical facts. Addressing these, alongside ensuring Fiber testnet reliability, is paramount for a successful MVP.

---

### (1) What specific technical facts are still UNVERIFIED for the MVP?

Here's a breakdown of the unverified technical facts, focusing on the requested areas:

*   **[HIGH RISK] RetroArch UDP RAM Reading Performance and Stability on Pi5**
    *   **What we assume:** RetroArch's UDP memory interface (`READ_CORE_MEMORY`) will be performant and stable enough for continuous polling (e.g., 60Hz) on the Pi5 (192.168.68.82) running the target OS.
    *   **What we actually know:** The `retroarch-memory-interface.md` and `fiberquest-gap-analysis-risks.md` findings confirm the existence of the `READ_CORE_MEMORY` command via UDP on port 55355, using hexadecimal addresses. However, there are no performance metrics, polling limits, or stability guarantees provided for continuous, high-frequency polling across different OS/hardware combinations, specifically the Pi5 (arm64). The `fiberquest-gap-analysis-risks.md` explicitly flags this as a HIGH RISK.

*   **[HIGH RISK] Fiber RPC JSON Format for `send_payment` and `open_channel`**
    *   **What we assume:** The Fiber Network Node (FNN) RPC methods `open_channel` and `send_payment` follow a standard JSON-RPC 2.0 format with clearly documented parameters and return types.
    *   **What we actually know:** The `fiber-rpc-api-reference.md` research explicitly states that the official RPC documentation (`docs/en/rpc.md` and `src/rpc/mod.rs`) was inaccessible (HTTP 404 errors) during crawling. While `fiber-trampoline-routing.md` mentions `trampoline_hops` for `send_payment`, the complete JSON structure, required parameters, and expected responses for both `open_channel` and `send_payment` are entirely unknown. This is a critical blocker identified as HIGH RISK in `fiberquest-gap-analysis-risks.md`.

*   **[HIGH RISK] Node.js Fiber Client Library Availability**
    *   **What we assume:** A readily available, official, or community-maintained Node.js client library exists to abstract interaction with the Fiber RPC, simplifying `open_channel` and `send_payment` calls.
    *   **What we actually know:** The research, particularly `fiberquest-gap-analysis-risks.md` and `fiberquest-web-client.md`, found no evidence of an official or widely adopted Node.js client library for Fiber. Direct RPC interaction would require manual construction of JSON-RPC requests over HTTP/WebSocket, which adds significant development overhead and error potential. This is a HIGH RISK.

*   **[HIGH RISK] Private Key Management for the Node.js Sidecar**
    *   **What we assume:** The Node.js sidecar can securely manage and use a private key to sign CKB transactions required for Fiber channel operations (opening, closing, potentially funding).
    *   **What we actually know:** The research (`fiberquest-gap-analysis-risks.md`, `fiberquest-hackathon-gap-analysis.md`) does not detail any specific, secure mechanism for the Node.js sidecar to manage or access private keys. While `ckb-ccc/core` is available for CKB transaction building, the integration of a private key for signing Fiber-specific channel transactions within the Node.js environment, without exposing the key, remains an unverified and critical security/operational fact. This is a HIGH RISK.

*   **[MEDIUM RISK] Testnet Fiber Channels Reliability for Demo**
    *   **What we assume:** Fiber testnet channels will be stable and reliable enough for a public hackathon demo, allowing smooth channel opening, payments, and closing.
    *   **What we actually know:** The `fiber-channel-funding-ux.md` and `MEMORY.md` explicitly state that the N100 Fiber node (`ckb1qzda0cr08m85hc8jlnfp3zer7xulejywt49kt2rr0vthywaa50xwsqg0dg5q5dpt0ww2exefvx4f68vffqa670gnhk9ej`) "NEEDS FUNDING (99+ CKB) to auto-accept channels" and that the channel between `ckbnode` and `N100` is currently "PENDING (N100 insufficient balance)". While this indicates a solvable setup issue rather than a fundamental protocol flaw, the practical reliability and stability of testnet channels for a live demo remain unverified until fully funded, opened, and tested under load. This is a MEDIUM RISK.

### (2) What are the 3 biggest things that could kill the project in the first 48 hours of building? For each: what's the mitigation / fallback plan?

1.  **Project Killer: Inability to interact with Fiber RPC (open_channel, send_payment)**
    *   **Why it kills:** This is the core payment mechanism. If we cannot reliably open channels or send payments, the entire "micropayments for game events" concept fails. The lack of documented RPC formats and a Node.js client library makes this a significant hurdle.
    *   **Mitigation / Fallback:**
        *   **Primary Mitigation:** Immediately prioritize reverse-engineering the Fiber RPC. Use `curl` or a simple Python script to interact directly with the `ckbnode` Fiber RPC (via N100:8237 SSH tunnel). Examine the `nervosnetwork/fiber` Rust source code (specifically `crates/fiber-lib/src/rpc/` and `src/rpc/`) for method signatures and data structures.
        *   **Fallback Plan (if RPC interaction remains blocked):** Pivot to a simpler "invoice-based" demo. The Node.js sidecar would generate a Fiber invoice (using a known `new_invoice` RPC call, which is more likely to be documented or inferable) and display it as a QR code or text. A human player would then manually scan/copy and pay the invoice using a separate Fiber wallet (e.g., a CLI tool or another funded Fiber node). This demonstrates Fiber payments but loses the real-time automation.

2.  **Project Killer: RetroArch UDP RAM Reading is Unreliable or Too Slow on Pi5**
    *   **Why it kills:** The entire game event detection relies on this. If the Node.js sidecar cannot consistently and quickly read game RAM, payment triggers for events like "HP lost" or "score change" will be inaccurate or fail entirely.
    *   **Mitigation / Fallback:**
        *   **Primary Mitigation:** Dedicate the first few hours to rigorous testing of the `READ_CORE_MEMORY` UDP command on the Pi5 with a simple game (e.g., Street Fighter II Turbo) and varying polling rates. Profile the Node.js sidecar's performance.
        *   **Fallback Plan (if high-frequency polling fails):** Reduce the scope of payment triggers. Instead of per-HP or per-score payments, implement simpler, less frequent triggers:
            *   **Game Start/End Payments:** A fixed "entry fee" at game start and a "winner's bonus" at game end. This only requires two reliable RAM reads per game.
            *   **Manual Triggers:** Implement a simple web UI for the Node.js sidecar with buttons like "Player 1 Wins Round," "Player 2 Takes Damage," allowing manual intervention for the demo.

3.  **Project Killer: Inability to Securely Sign Fiber Channel Transactions from Node.js Sidecar**
    *   **Why it kills:** Fiber channels require CKB on-chain transactions for opening and closing. If the Node.js sidecar cannot sign these transactions, channels cannot be established, and the entire Fiber integration is blocked. Exposing private keys in a hackathon demo is unacceptable.
    *   **Mitigation / Fallback:**
        *   **Primary Mitigation:** For the hackathon MVP, use a **dedicated, pre-funded testnet wallet with a hardcoded private key** within the Node.js sidecar (clearly labeled as "DEMO ONLY - NOT FOR PRODUCTION"). This allows functional testing of channel operations without solving the full security challenge. Leverage `ckb-ccc/core` for transaction construction and signing with this temporary key.
        *   **Fallback Plan (if even hardcoded key signing is problematic):** Pre-open a set of Fiber channels manually using the `fnn` CLI on `ckbnode` and `N100` before the hackathon. The Node.js sidecar would then only need to send payments within these already-open channels, avoiding the need for on-chain signing during the demo. This limits the demo to existing channels but ensures payments can flow.

### (3) Exact day-1 checklist for March 11: the first 5 things to do when hackathon opens, in order, with expected time per task.

**Goal for Day 1:** Establish fundamental communication channels and verify core payment mechanics.

1.  **Verify RetroArch UDP RAM Reading on Pi5 (192.168.68.82)**
    *   **Action:** Install RetroArch on the Pi5, load Street Fighter II Turbo (SNES core `snes9x`). Write a minimal Node.js script to send `READ_CORE_MEMORY` commands to `127.0.0.1:55355` (or Pi5's LAN IP if sidecar is on another machine) for P1 Health (`0x0530`) and P2 Health (`0x0730`). Continuously poll at ~60Hz and log responses. Observe stability, latency, and parseability of data.
    *   **Expected Time:** 2 hours.

2.  **Fund N100 Fiber Node & Establish `ckbnode`↔`N100` Channel**
    *   **Action:** Transfer 100 CKB to N100's Fiber wallet (`ckb1qzda0cr08m85hc8jlnfp3zer7xulejywt49kt2rr0vthywaa50xwsqg0dg5q5dpt0ww2exefvx4f68vffqa670gnhk9ej`). On `ckbnode` (via SSH tunnel N100:8237 to `ckbnode:127.0.0.1:8227`), use the `fnn open_channel` RPC command, targeting N100's PeerId (`QmTh1V2gHqXKs59sGL24XpRHFGLch4J6wT4sdGB7EhRgAm`). Verify channel status using `fnn list_channels` on both nodes.
    *   **Expected Time:** 1 hour (plus CKB transaction confirmation time, which can be monitored in parallel).

3.  **Reverse-Engineer Fiber RPC `open_channel` & `send_payment` JSON Formats**
    *   **Action:** Using `curl` or a simple Python script, interact directly with `ckbnode`'s Fiber RPC (via SSH tunnel N100:8237). Attempt to call `open_channel` and `send_payment` with educated guesses for parameters, referencing `fiber-trampoline-routing.md` for `send_payment` hints (e.g., `trampoline_hops`). Simultaneously, inspect the `nervosnetwork/fiber` Rust source code for RPC definitions if direct calls are failing.
    *   **Expected Time:** 3 hours.

4.  **Implement Basic Node.js Fiber RPC Client**
    *   **Action:** Based on the reverse-engineered JSON formats from Task 3, write a lightweight Node.js module. This module should use `axios` or `node-fetch` to make HTTP POST requests to the Fiber RPC endpoint. Implement functions for `open_channel` and `send_payment` (and potentially `new_invoice`, `list_channels`). For signing, integrate a placeholder using a hardcoded testnet private key (as per mitigation for Project Killer 3).
    *   **Expected Time:** 4 hours.

5.  **Connect RetroArch Sidecar to Fiber Client for Game Start/End Payments**
    *   **Action:** Integrate the newly created Node.js Fiber client into the RetroArch sidecar. Modify the sidecar to detect a "game start" event (e.g., RetroArch `GET_STATUS` showing a game loaded) and trigger an `open_channel` call. Detect a "game end" event (e.g., `GET_STATUS` showing no game, or a specific RAM address change) and trigger a `send_payment` (e.g., a small "exit bonus" or "channel settlement" payment). This establishes the most basic end-to-end payment flow.
    *   **Expected Time:** 4 hours.
