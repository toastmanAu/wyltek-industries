# Research: esp32p4-emulator-payment-synthesis

**Date:** 2026-03-05  
**Status:** SYNTHESIS  
**Seeds:** local findings + MEMORY.md

---

As a technical architect and project advisor, I've synthesized the provided research findings and project ground truth for FiberQuest. The project leverages a robust CKB foundation, but critical integration and verification steps remain for the hackathon MVP.

### Executive Summary

FiberQuest is an ambitious hackathon project aiming to integrate retro gaming with Nervos Fiber micropayments, potentially on a self-contained ESP32-P4 device. The core infrastructure, including operational Fiber nodes on `ckbnode` and `N100`, CKBFS for content, and the DOB Minter, is largely functional. Crucially, specific game RAM addresses have been identified for key titles, enabling direct game state interaction. While a self-contained ESP32-P4 device is technically feasible, it presents significant challenges in resource management and requires a "signer-only" approach for Fiber. The biggest accelerator would be reliable access to RetroAchievements RAM maps, which currently faces programmatic access issues.

---

### (1) Is a self-contained FiberQuest device on ESP32-P4 (emulator + signer + WiFi payment) technically feasible? Give a concrete yes/no with reasoning.

**Yes, it is technically feasible, but with significant caveats and a focused scope.**

**Reasoning:**

1.  **Emulator Viability:** The `esp32p4-snes-emulator-viability.md` finding confirms that ESP32-P4 is viable for NES and likely SNES emulation due to its superior specifications (dual-core 400MHz RISC-V CPU, ample clock speed, MIPI DSI for display). Existing NES emulators target ESP32-P4, and SNES is expected to run given its capabilities.
2.  **Dual-Core Advantage:** The `fiberquest-esp32p4-capabilities.md` explicitly states the ESP32-P4 has a "dual-core 400MHz RISC-V CPU." This is crucial, as it allows for dedicated task separation: one core for the demanding real-time emulator, and the other for background tasks like WiFi, Fiber signing, and game event detection. This directly addresses the "single-core" concern raised in `esp32p4-snes-emulator-viability.md`.
3.  **Fiber Signing:** The `esp32p4-fiber-signer-vs-lightclient.md` research indicates that a "Signer Only" approach for Fiber (secure private key storage and signing of payment hashes) is feasible and minimizes resource footprint. Running a full Rust-based Fiber Network Node (FNN) or a comprehensive CKB light client on the ESP32-P4 is not currently supported and likely too resource-intensive given the lack of specific benchmarks.
4.  **Memory:** The ESP32-P4's 768KB on-chip SRAM combined with support for large external PSRAM (e.g., 8MB or 16MB) provides sufficient memory for an emulator, game ROMs, and the minimal Fiber signing logic.
5.  **Connectivity:** Integrated WiFi is standard, and USB 2.0 HS OTG allows for modern controller input.

**Caveats:** The feasibility hinges on a "Signer Only" Fiber integration, careful FreeRTOS task prioritization, and efficient C/C++ implementations for all components to manage the constrained environment.

### (2) Best architecture for a standalone device: which tasks run on which core, what's the memory budget, what gets cut if RAM is tight.

**Best Architecture for ESP32-P4 Standalone Device:**

*   **Core 0 (High-Performance RISC-V):**
    *   **Primary Task:** Running the retro game emulator (NES/SNES core). This requires consistent, low-latency CPU cycles for game logic, audio, and video rendering.
    *   **Secondary Task:** Direct controller input polling (if using GPIO-based retro controllers) to minimize latency.
*   **Core 1 (Secondary RISC-V):**
    *   **Primary Task:** WiFi stack management and network communication.
    *   **Secondary Tasks:**
        *   **Fiber Signer:** Secure private key storage and `secp256k1` signing of Fiber payment hashes and CKB L1 transaction components (for channel open/close).
        *   **Game Event Detection:** Polling RetroArch's core memory (via an internal mechanism or shared memory) to detect in-game events (e.g., health changes, score updates).
        *   **Payment Trigger Logic:** Executing the Node.js sidecar's logic (re-implemented in C/C++ or a lightweight scripting engine) to determine when to trigger Fiber payments.
        *   **Display Output Management:** Handling MIPI DSI display buffer updates and synchronization with Core 0's rendering.
        *   **USB Host Stack:** For modern USB controllers.

**Memory Budget (with external PSRAM):**

*   **On-chip SRAM (768KB):** Critical for real-time tasks, CPU caches, and core FreeRTOS operations.
*   **External PSRAM (e.g., 8MB or 16MB):** Essential for:
    *   **Emulator:** Game ROMs, framebuffer, emulator state (1-4MB depending on console/game).
    *   **Fiber Signer/Minimal CKB Logic:** Code and data for `libsecp256k1`, transaction serialization (likely <1MB).
    *   **WiFi Stack:** Network buffers, connection state (hundreds of KB).
    *   **Game Event Data:** RAM maps, current game state (tens of KB).
    *   **Total:** An 8MB PSRAM module should provide sufficient headroom for this architecture.

**What gets cut if RAM is tight:**

1.  **CKB Light Client:** The most significant cut. If PSRAM is limited or absent, the ESP32-P4 must operate purely as a "Signer Only" device, relying entirely on a remote CKB node (e.g., `ckbnode` or `N100`) for all on-chain state verification and transaction construction.
2.  **Number of Supported Games/ROMs:** Reduce the number of game ROMs stored on flash and loaded into RAM. Only one game at a time.
3.  **Emulator Features:** Disable non-essential emulator features (e.g., savestates, rewind, advanced video filters) to reduce memory footprint.
4.  **UI/Menu System:** Simplify the on-device user interface to a bare minimum, potentially offloading complex menus to a web-based companion app.
5.  **Payment Logic Complexity:** Reduce the granularity or complexity of in-game payment triggers, focusing on simpler events (e.g., round win/loss instead of per-HP damage).

### (3) If RetroAchievements rich presence / achievement data gives us pre-verified RAM maps, how does this change our development speed? Quantify: instead of manually finding RAM addresses for 5 games, we get community-verified maps for 1000s of games instantly.

The availability of pre-verified RAM maps from RetroAchievements (RA) rich presence and achievement data would **massively accelerate development speed, shifting the bottleneck from discovery to integration.**

**Quantification:**

*   **Manual RAM Mapping (Current State):** Manually finding RAM addresses for critical game states (health, score, lives) for a single game can be a **highly time-consuming and tedious process, often taking days to weeks per game** for an experienced reverse engineer. This involves using memory scanners, playing the game repeatedly, and trial-and-error to isolate dynamic memory locations. For 5 games, this could easily consume **1-2 weeks of dedicated developer time.**
*   **RetroAchievements Data (Ideal State):** If we could reliably access and parse RA data, which explicitly defines memory locations and logic (as noted in `retroachievements-live-api-payment-triggers.md`), the process would be transformed. Instead of discovery, it becomes **parsing and integration**. For 1000s of games, this could mean:
    *   **Initial Setup:** A few days to develop a robust parser for RA data (if API access is granted/fixed).
    *   **Per-Game Integration:** Once the parser is built, integrating a new game's RAM map could be **reduced to minutes or hours**, depending on the complexity of the payment trigger logic for that specific game.
*   **Impact:** This would allow FiberQuest to scale from a handful of manually mapped games to a **vast, community-curated catalog of Fiber-enabled games almost instantly.** It would enable a much richer, more diverse demo and product offering, significantly enhancing the project's appeal and long-term viability. The current research findings (`retroachievements-live-api-payment-triggers.md`, `retroachievements-ram-maps.md`, `fiberquest-target-games.md`) indicate that programmatic access to this data is currently problematic (403/404 errors), making this a high-impact research task.

### (4) The boldest possible demo: a single ESP32-P4 device running NES/SNES, connected to a TV via HDMI/composite, two controllers plugged in, WiFi connected to Fiber, payments triggered by in-game events — is this achievable in the 2-week hackathon? What's the minimum hardware BOM?

**Achievability in 2-week hackathon:** **Yes, but with significant scope reduction and careful prioritization.**

The core functionality (emulator + signer + payments) is achievable. The "boldest" aspect of HDMI/composite output is the highest risk and likely requires simplification or an external adapter.

**Reasoning:**

*   **Emulator + Payments:** As established in Q1, running NES/SNES emulation on ESP32-P4 with a "Signer Only" Fiber integration and WiFi is technically feasible. The game event detection and payment triggering logic (based on pre-identified RAM addresses like those in `fiberquest-game-catalog.md`) are the core of the hackathon.
*   **Controllers:** The ESP32-P4's USB 2.0 HS OTG and ample GPIOs make connecting two controllers (either modern USB or retro via custom adapters) achievable.
*   **TV Output (Highest Risk):**
    *   **MIPI DSI:** The ESP32-P4 has MIPI DSI, which is suitable for direct display connection. Using a small DSI panel (e.g., 4-7 inch) is the most realistic and achievable display output for a hackathon.
    *   **HDMI/Composite:** Implementing HDMI or composite output directly from the ESP32-P4 within two weeks is extremely challenging. It would likely require an external HDMI/composite encoder chip with a custom driver, or a pre-made adapter solution, adding complexity and potential integration hurdles. For the hackathon, a direct DSI display is a safer bet for a "self-contained" device. If a TV is a hard requirement, an external HDMI adapter (e.g., DSI-to-HDMI) would be necessary, but this adds cost and another point of failure.

**Minimum Hardware BOM for a "Boldest Achievable" Demo (prioritizing DSI display for reliability):**

*   **ESP32-P4 Development Board:** (e.g., an official Espressif board or a third-party module with integrated 8MB/16MB PSRAM and MicroSD slot).
*   **MicroSD Card:** 16GB (for firmware and game ROMs).
*   **USB-C Power Supply:** 5V, 2A minimum.
*   **USB Hub:** (if using multiple USB controllers, not strictly necessary if dev board has multiple USB host ports or GPIO for retro controllers).
*   **USB Game Controllers (2x):** Standard USB HID gamepads (e.g., Xbox 360/One, PS4/5, or generic USB controllers).
*   **Small MIPI DSI Display:** 4-inch to 7-inch DSI panel (e.g., 800x480 or 1024x600 resolution).
*   **Optional (for HDMI TV):** A DSI-to-HDMI adapter board (if available and compatible), but this is a significant stretch.
