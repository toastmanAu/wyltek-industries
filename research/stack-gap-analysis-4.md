## Executive Summary

The Wyltek Industries stack demonstrates remarkable progress, with core Nervos CKB infrastructure, embedded systems, and a production-ready DOB minting platform already shipped. Phill's deep hardware expertise and proactive approach to local LLM inference are significant assets. While the foundation is robust, recent research has highlighted critical gaps in CKB DID understanding, secure OTA update mechanisms, comprehensive Fiber Network tooling for Node.js, and detailed LLM cost optimization. Addressing these will enhance reliability, unblock key planned features, and refine operational efficiency across the Wyltek ecosystem.

## Gap Analysis

### (1) What parts of the Wyltek stack are well-researched and ready to build.

The Wyltek stack has several mature and well-understood components, ready for further development or already in production:

*   **CKB Light Client on ESP32 (`ckb-light-esp`):** This is a fully functional and performant CKB light client for ESP32, confirmed to run efficiently on ESP32-P4 with negligible CPU impact. Its core protocol stack is robust and tested. The `bitchat_mesh` component within it is also well-defined in its initial stages.
*   **CKB Eaglesong Mining on ESP32 (`NerdMiner CKB`):** The core Eaglesong mining functionality and Stratum protocol implementation are complete and shipped. The Telegram OTA feature is merged and flashed, awaiting final WiFi configuration.
*   **CKB Stratum Proxy (`ckb-stratum-proxy`):** This Node.js proxy is fully functional and running in production, handling ViaBTC quirks and extranonce allocation.
*   **DOB Minting System (`ckb-dob-minter` & `@wyltek/ckbfs-browser`):** The React/Vite DOB minting application is live in production, leveraging the browser-side CKBFS V3 SDK for image uploads. The JSON schema for hardware provenance DOBs is also well-defined (`dob-hardware-provenance.md`).
*   **Embedded CKB Framework (`wyltek-embedded-builder`):** A C framework for ESP32 CKB/blockchain apps exists, providing board targets and sensor drivers.
*   **CKB Node Monitoring (`ckb-node-dashboard` & `ckb-whale-bot`):** Both the Node.js proxy/HTML dashboard and the Telegram whale alert bot are running in production, providing valuable monitoring capabilities.
*   **Wyltek Industries Site:** The main static site is live, with a functional JoyID-based member system and a bug reporter (WIP, but code is written).
*   **Fiber Network Infrastructure:** Our `ckbnode` runs a funded Fiber node, and `N100` runs an unfunded one. The core concepts of Fiber (payment channels, low latency, PTLCs) are well understood.
*   **ESP32 CKB Signing:** `secp256k1` signing on ESP32 is confirmed working and is the recommended primitive for CKB transactions (`esp32-ckb-dob-signing-flow.md`).
*   **LLM Cost Optimization Strategy:** The general strategy using LiteLLM for routing and Ollama for local inference is understood, with some pricing data collected (`llm-cost-optimisation.md`).
*   **Local Repository Mirroring:** The strategy for mirroring key Nervos/CKB repositories locally for faster AI-assisted code search is clear and feasible (`local-repo-mirror.md`).

### (2) What gaps remain — things we know we need but haven't researched.

These are areas where Wyltek has a clear need or a planned feature, but the underlying technical details are either completely unresearched or the research attempts failed to yield answers:

*   **CKB DID Cell Format and Contract:** The `WyDID.h` component is planned, but `ckb-did-cell-format.md` indicates a complete lack of understanding of the on-chain `did:ckb` cell structure, the validating smart contract, and the derivation from key to DID string. This is a fundamental identity primitive.
*   **CKBFS V3 Protocol Details:** While `@wyltek/ckbfs-browser` exists and the DOB minter uses CKBFS V3, `ckbfs-v3-vs-v2-migration.md` shows all research attempts to understand the specifics of V3 (witness structure, molecule schema, cell layout, cost, migration path) failed. This is crucial for deeper integration or future embedded CKBFS applications.
*   **Secure ESP32 OTA with Rollback:** `telegram-bot-esp32-security-hardening.md` revealed a critical security vulnerability in FastBot (no SSL) for Telegram OTA. Furthermore, `esp32-wifi-ota-partition-safety.md` confirmed that the current `huge_app.csv` partition scheme is insufficient for safe OTA with rollback. This is a high-priority security and reliability gap for all ESP32 products.
*   **Cloudflare Workers KV for Rate Limiting:** The `Cloudflare Bug Reporter Worker` is in progress, but `cloudflare-workers-kv-rate-limiting.md` indicates a lack of research on how to effectively use Cloudflare Workers KV for rate limiting, including `wrangler.toml` configuration, atomic increment patterns, and free tier limits. This impacts the robustness of our serverless functions.
*   **ESP32-P4 Post-Quantum Cryptography (SPHINCS+):** `esp32-p4-sphincs-plus-revisit.md` and `esp32-p4-sphincs-plus.md` show complete failure to gather information on the feasibility, performance, and ESP-IDF integration of SPHINCS+ on ESP32-P4. This is a planned future-proofing measure.
*   **Node.js Fiber Client Library:** `fiber-node-js-client-library-search.md` explicitly states there is no official or community-maintained Node.js/TypeScript client library for Fiber RPC. This is a critical infrastructure gap for any Node.js application (like `ckb-stratum-proxy`, `ckb-node-dashboard`, or future agent services) that needs to interact with Fiber. While `fiber-nodejs-client-library-development.md` infers method signatures from Rust source, a concrete, usable library is missing.
*   **Fiber RPC API Reference:** `fiber-rpc-api-reference.md` and `fiber-rpc-api-verification.md` indicate that a complete, verifiable API reference for Fiber RPC (methods, parameters, return types, authentication, protocols) is still largely unknown due to inaccessible documentation. While `fiber-rpc-raw-source.md` and `fiber-rpc-json-format-details.md` extract some details from Rust source, a comprehensive, verified reference is missing.
*   **LLM Cost Optimization Details:** While a strategy exists, `llm-cost-optimisation-pricing-update.md` and `llm-cost-optimisation.md` highlight missing specific pricing for some models (GPT-4o mini, Llama 3.3 70B free tier), detailed RPM/TPD limits for free tiers, and practical routing recommendations for different task types. This impacts the efficiency of our agent collective.
*   **Renault Clio RS 172 OBD2/CANBus Integration:** `obd2-canbus-esp32.md` and `obd2-clio-rs172-esp32-revisit.md` show a complete lack of specific vehicle information (bus type, ECU, K-Line protocol specifics, data read/write capabilities) for the planned automotive integration.
*   **NerdMiner CKB Pool Stats Display:** `nerdminer-ckb-pool-stats-display.md` indicates that ViaBTC's public API for per-worker stats is inaccessible, and the NerdMiner CKB currently doesn't parse pool responses for accepted/rejected shares. This limits the utility of the miner's display.
*   **Spore DOB Rendering Standards:** `spore-dob-rendering-standards.md` reveals a gap in understanding higher-level metadata standards for Spore DOBs and how wallets/explorers render `ckbfs://` URIs. This affects the discoverability and presentation of our minted DOBs.
*   **Wyltek Membership Social Layer:** `wyltek-membership-ckb-dob-social-layer.md` shows no research on advanced social features like soulbound membership tokens, on-chain comments, post-mint metadata updates, or native CKB token-gated architectures. This limits the potential for community engagement.
*   **Wyltek Site SEO Optimization:** `wyltek-seo-current-audit.md` identified an invalid `sitemap.xml` and missing essential meta tags. `wyltek-site-seo.md` indicates a lack of research on integrating with Nervos ecosystem directories and community channels for discoverability.
*   **Supabase JWT Generation from CKB Addresses:** `supabase-rls-member-security-audit.md` points to a gap in the specific flow for generating Supabase JWTs from CKB addresses, which is crucial for enhancing the security of our member system.

### (3) What new research tasks should be queued based on recent work (bug reporter, NerdMiner OTA, research page improvements, Telegram OTA, DOB minting system).

Based on the recent work and identified gaps:

*   **Bug Reporter:** The Cloudflare Bug Reporter Worker is WIP. The `cloudflare-workers-kv-rate-limiting.md` finding highlights a need for robust rate limiting.
*   **NerdMiner OTA / Telegram OTA:** The Telegram OTA for NerdMiner CKB is WIP. `telegram-bot-esp32-security-hardening.md` identified a critical security flaw (no SSL), and `esp32-wifi-ota-partition-safety.md` identified a partition scheme issue.
*   **Research Page Improvements / Wyltek Site:** The Wyltek site has a research page. `wyltek-seo-current-audit.md` and `wyltek-site-seo.md` identified significant SEO gaps.
*   **DOB Minting System:** The DOB minter is live. `spore-dob-rendering-standards.md` and `wyltek-membership-ckb-dob-social-layer.md` identified gaps in rendering and social features.

### (4) Any dependencies between planned features that could cause build order issues.

Several critical dependencies and blockers exist:

*   **N100 Fiber Node Funding:** The `N100 fiber` node "NEEDS FUNDING (99+ CKB)" (`MEMORY.md`). This is a direct blocker for any Fiber-related development or testing that relies on the N100 node, including establishing channels for FiberQuest or any future agent collective Fiber interactions.
*   **OPi5+ Ollama PSU Upgrade:** The `qwen2.5:7b` model causes power-related crashes on the Orange Pi 5+ (`MEMORY.md`). This is a blocker for stable local LLM inference on the OPi5+, impacting the reliability and capability of the agent collective.
*   **Agent Collective HuggingFace Token:** The deployment of the Wyltek agent on the N100 is "blocked by the lack of a HuggingFace token from Phill" (`MEMORY.md`). This prevents the agent from accessing free HF models.
*   **Cloudflare Bug Reporter Worker Deployment:** The worker is coded but "Needs CF login + wrangler deploy" (`STACK.md`). This is a manual deployment blocker.
*   **Telegram OTA WiFi Configuration:** The Telegram OTA for NerdMiner CKB is flashed but "Waiting on: WiFi config via NerdMinerAP" (`STACK.md`). This prevents full testing and activation of the feature.
*   **Node.js Fiber Client Library:** The absence of an official Node.js Fiber client library (`fiber-node-js-client-library-search.md`) is a critical dependency for *any* Node.js application that needs to interact with the Fiber Network. This impacts FiberQuest directly, but also any future Node.js-based Wyltek services that might leverage Fiber. This needs to be built or a robust wrapper developed before extensive Node.js Fiber integration.
*   **CKB DID Research:** The `WyDID.h` component is planned, but the fundamental research on `did:ckb` cell format and contract is missing. This research is a prerequisite for any robust on-device identity solution.
*   **Secure ESP32 OTA:** The identified security and partition issues for Telegram OTA are critical. Addressing these is a prerequisite for safely deploying future OTA updates to NerdMiner CKB and other ESP32 devices.
