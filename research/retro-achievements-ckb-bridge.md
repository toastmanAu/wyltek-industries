## Research Topic: retro-achievements-ckb-bridge

## Summary
The `rcheevos` library provides C code for emulators to process RetroAchievements data, including achievement and leaderboard support. It explicitly states it does not handle HTTP network connections; clients are responsible for fetching data from RetroAchievements and submitting results. The `rapi` component helps clients build correct URLs for server communication, and `rc_client_t` functions offer callbacks for client-side functionality, including UI and HTTP interactions. While the specific achievement definition format is not detailed, the architecture suggests that an emulator integrating `rcheevos` could be modified to route achievement unlock events to a custom CKB destination, potentially alongside or instead of the official RetroAchievements server.

## Questions to Answer

### 1. RetroAchievements API: what endpoints exist for reading achievement definitions and submitting unlocks? Can we POST to a custom endpoint instead?
The provided content indicates that the `rapi` component (headers `rc_api_user.h`, `rc_api_runtime.h`, `rc_api_common.h`) is used to "build URLs to access many RetroAchievements web services." An example `rc_api_init_login_request` is cited, suggesting functions exist to prepare requests for user and runtime interactions, which would logically include submitting unlocks and potentially reading definitions. However, `rcheevos` itself "does *not* provide HTTP network connections." Clients are responsible for making the actual HTTP requests using the URLs generated by `rapi` and then passing the server's response back to `rcheevos` for processing.

Regarding posting to a custom endpoint: Since the client (emulator) is explicitly responsible for making the HTTP requests, it *can* be designed to POST to a custom endpoint instead of, or in addition to, the official RetroAchievements server. The `rcheevos` library merely provides the data and URL-building utilities; it does not enforce the destination of the HTTP request.

### 2. rcheevos library: what's the callback interface when an achievement fires? How hard is it to add a second callback that signs a CKB tx?
The `rc_client_t` functions provide "several callback functions which allow the client to implement dependent functionality (UI and HTTP calls)." The documentation points to `https://github.com/RetroAchievements/rcheevos/wiki/rc_client-integration` for details on these callbacks.

Without direct access to the `rc_client-integration` wiki page or the `rcheevos` source code, the exact signature of the callback interface when an achievement fires cannot be precisely determined from the provided content. However, the mention of "HTTP calls" as a dependent functionality suggests a mechanism for the client to be notified of events that require server interaction, such as an achievement unlock.

Adding a second callback that signs a CKB transaction would depend on the flexibility of the `rc_client_t` callback mechanism. If it allows for multiple handlers or if the existing HTTP callback can be extended, it should be feasible. The difficulty would primarily lie in integrating a CKB SDK (likely in C or a language that can interface with C) into the client application (emulator) to perform the transaction signing and submission, rather than modifying `rcheevos` itself. `rcheevos` is a C library, so integrating CKB signing logic would involve C-compatible libraries or external process calls.

### 5. RA has a community of 50,000+ users and 500,000+ achievements across thousands of games. Is there a path to propose CKB as an optional "export" destination for existing achievements?
Yes, there appears to be a technical path to propose CKB as an optional "export" destination, primarily by targeting the client (emulator) integration layer.

1.  **Client-side Control:** The `rcheevos` library explicitly states it "does *not* provide HTTP network connections" and that "Clients must get data from RetroAchievements, and pass the response down to **rcheevos** for processing." This means the emulator (the client) is responsible for all network communication.
2.  **`rc_client_t` Callbacks:** The `rc_client_t` functions provide "callback functions which allow the client to implement dependent functionality (UI and HTTP calls)." This is the crucial integration point. When an achievement fires and `rcheevos` notifies the client, the client's HTTP callback could be extended to:
    *   Submit the achievement unlock to the official RetroAchievements server.
    *   *Additionally* (or optionally instead), construct and sign a CKB transaction for the achievement unlock.
3.  **`RAIntegration` DLL:** The `RAIntegration` DLL is described as "The DLL used for interfacing with RetroAchievements.org." This suggests a common, perhaps standardized, integration point for emulators. Modifying this DLL or providing an alternative/wrapper that incorporates CKB transaction logic would be a direct way to implement this "export" functionality for emulators that use it.

The path would involve developing a client-side modification (e.g., a fork of `RAIntegration` or a new integration layer for emulators) that leverages the existing `rcheevos` callbacks to trigger CKB transactions. Proposing this to the RetroAchievements community would then be a matter of demonstrating its value and ease of integration.
